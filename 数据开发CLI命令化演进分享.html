02 Overview

背景定位：把平台能力变成可编排命令

大数据作业开发的标准流程包括登录平台、查询资源、创建作业、保存草稿、草稿调试、发布等流程。DIP CLI 的核心价值是把这些高频操作抽象成稳定命令，降低重复成本，并为脚本、Agent 和 CI 提供入口。

Training Focus

减少页面跳转

把业务组、目录、资源组、任务详情等查询动作收敛到终端。

降低误操作

通过参数校验、输出协议等减少非法数据入库风险。

沉淀团队经验

项目级配置和命令模板可以进入仓库，快速复用标准流程。

连接自动化

结构化输出能被脚本、 Agent 消费，推动开发流程自动化。

03 Value

提升价值：效率、准确性和自动化能力

工具功能列表 & CLI 带来开发方式变化。

User Benefit

开发效率提升

高频动作可复制、可别名化、可脚本化，减少重复点击和参数查找。

交付准确性提升

统一读取环境、空间、资源默认值，配合错误分类和 exit code 做前置校验。

自动化能力提升

-o json 和 --fields 将结果变成机器可读数据，方便 Agent 和 CI 编排。

04 Architecture

系统架构：本地轻量客户端，平台承载业务状态

DIP CLI 不替代数据中台，而是平台能力的本地命令入口。它向上承接开发用户、脚本和 Agent，向下调用 DIP 开发平台、开发中心、配置管理、权限、运维等服务API。

09 Practices

最佳实践：把个人习惯沉淀为团队资产

落地建议要具体到仓库、脚本、README 和调试规范。

Team Standard
维护项目级 config.yaml

把团队共享的空间、业务组、目录、资源组配置放进项目仓库，减少口口相传。

沉淀命令模板

在 README 中写清创建、调试、状态查询和发布命令，让新人直接复用。

查询类命令优先 JSON

脚本和 Agent 场景使用 -o json 和 --fields，避免解析 table 文本。

发布前先调试验证

建议团队统一“创建草稿 → 提交调试 → 查询状态 → 发布”的流程，降低误发布。

10 Roadmap

后续规划：从 CLI 工具到自动化底座

DIP CLI 的演进方向是继续扩展任务类型、增强诊断能力、支持元数据表导入、自动创建 Hive/Paimon/StarRocks 表，并深化 Agent/CI 集成。

Next

扩展任务类型

实时：FlinkSQL、PyFlink、 实时集成(kafka -> hive、tidb、starrocks..)
离线：SparkSQL、PySpark、算子处理

增强调试诊断

围绕作业调试、周期运行等场景运行日志分析失败原因输出诊断建议。

完善作业运维

集成运维中心相关能力，完善作业上下线、资源配置修改， 离线作业补数、实时作业运维等。

元数据表导入

支持从业务表结构导入、Hive/SR/Paimon表创建，减少手工配置。

自动创建湖仓表

基于导入元数据自动生成 Hive、Paimon、StarRocks 建表动作和校验结果。

支持批量操作

支持多任务、多表、多Topic 的批量查询和校验。

智能化配置

根据作业依赖、空间资源使用分布自动添加依赖、配置资源队列及调度周期。

深化 Agent 集成

用稳定输出和命令模板支撑智能体安全执行平台操作。
